# Full Clause Search Change Log

Date: 2026-04-10

## What This Change Does

This update changes semantic search so that:

- **embedding is now generated from full clause text**
- **retrieval uses full clause text for matching**
- **short clause summaries are still kept for compact display**

In simple terms:

- before: search mostly worked on compressed clause summaries
- now: search works on the real clause language underneath

## Why This Change Was Needed

The earlier pipeline shortened clauses too early.

That caused three search problems:

1. semantic meaning was lost before vector indexing
2. lexical overlap used summary text instead of the original clause language
3. precedent-style retrieval could miss relevant matches because the indexed text was too small

## Files Changed

## 1. `ML-model-main/ml-service/app/schemas.py`

### What changed

Updated the ML response schema so each clause can now carry:

- `clause_text`
- `clause_text_full`
- `clause_text_summary`

### Why

The ML service now needs to return both:

- the full original clause text
- the shortened display summary

---

## 2. `ML-model-main/ml-service/app/predictor.py`

### What changed

Changed the clause payload generated by the Python analyzer.

Before, each clause mostly returned only:

- `clause_text` as the shortened summary

Now each clause returns:

- `clause_text` as the summary for backward compatibility
- `clause_text_full` as the normalized full clause text
- `clause_text_summary` as the short clause summary
- `extracted_values`

### Why

This is the first place where the original clause meaning was being collapsed.

Now the backend can index and search the full clause text instead of only the shorter summary.

---

## 3. `backend/services/mlAnalysis.service.js`

### What changed

Updated both analysis paths:

- Python ML response normalization
- Node heuristic fallback

New internal clause shape now includes:

- `clauseText`
- `clauseTextFull`
- `clauseTextSummary`

Also added a clause-body normalizer so full text is stored with clean whitespace.

### Why

The backend needs a stable internal contract clause shape whether the clause comes from:

- Python ML
- or Node fallback

Without this, full-text retrieval would only work in one analysis path and break in the other.

---

## 4. `backend/services/contract.helpers.js`

### What changed

Updated `buildClauseRecords(...)` so stored clause records now include:

- `clauseText` as the short summary for backward compatibility
- `clauseTextFull`
- `clauseTextSummary`

Updated risk summaries to prefer:

- `clauseTextSummary`

### Why

This makes the stored contract data support both:

- compact UI display
- full-text semantic retrieval

---

## 5. `backend/services/contract.service.js`

### What changed

Updated vector creation and clause-precedent lookup logic.

#### Vector indexing

Before:

- embeddings were created from `clause.clauseText`

Now:

- embeddings are created from `clause.clauseTextFull || clause.clauseText`

#### Vector metadata

Added:

- `clauseTextSummary`
- `clauseTextFull`

#### Clause-level precedent lookup

Updated clause insight retrieval so it embeds and queries using:

- full clause text

### Why

This is the core switch that makes semantic search actually use the full clause language.

---

## 6. `backend/services/vector.service.js`

### What changed

Updated local retrieval scoring so lexical overlap now uses:

- `metadata.clauseTextFull`
- then `metadata.clauseTextSummary`
- then legacy `metadata.clauseText`

### Why

Even when Pinecone is unavailable and the app falls back to local vector search, the retrieval logic should still use the full clause language.

This keeps search behavior consistent in both live and fallback modes.

---

## 7. `backend/services/insight.service.js`

### What changed

Updated the reasoning layer to understand both summary and full clause text.

#### Supporting matches

Now returns:

- summary text for compact result display
- full clause text alongside it

#### Prompt context for reasoning

Gemini/template prompt context now includes full clause text for:

- precedent matches
- clause insight generation
- contract overview target clauses

### Why

Better retrieval is only fully useful if the answer layer is grounded in the same richer clause evidence.

## What Did Not Change

These parts were intentionally left unchanged in this step:

- frontend contract cards
- search page layout
- risk board display
- document search route
- embedding provider itself

Reason:

This step was focused only on making semantic search use full clause text without changing the current UI contract.

## Backward Compatibility

This change keeps `clauseText` in the stored clause record so the current UI does not break.

That means:

- old UI components can still read `clauseText`
- new search logic can use `clauseTextFull`

## Important Rollout Note

This change improves the pipeline for **newly ingested or reindexed contracts**.

If you already have contracts indexed using summary-only clause text, those old vectors will still reflect the old behavior until you:

1. restart the Python ML service
2. re-upload or reindex the contracts

Without reindexing, existing search results may still reflect the older compressed-clause vector data.

## Recommended Next Step

To get the full benefit of this change, the next step should be:

1. create a reindex script for existing contracts
2. then replace hash embeddings with a real embedding provider

That combination will improve semantic search much more than prompt tuning alone.
